# FreeBSD 状态报告中文翻译格式规范

本规范用于统一全书 102 份状态报告（2001-2026）的格式、风格、术语与排版。本规范与 [CLAUDE.md](CLAUDE.md) 并行，二者均为有效规范；若同一事项两文档表述不同，以 CLAUDE.md 为准。关键格式规则（CJK 空格、路径加粗、man 页引用格式、命令格式、引号规范等）见下方第十章术语表与第十七章格式规范。

## 一、年代阶段划分

| 阶段 | 年代范围 | 文件名模式 | 上游源格式 | 主要特征 |
| ---- | -------- | ---------- | ---------- | -------- |
| **早期** | 2001-2009 | 月份范围 `1-3.md` | SVN XML | 元信息缺"版本"项；裸 URL 多 |
| **中期** | 2010-2016 | 季度 `Q1.md` | SVN XML | "原文链接"与"原文"混用；`**链接**` 加粗 |
| **过渡期** | 2017-2020 | 混合 | SVN XML | 元信息趋于稳定；URL 后缀 `.html` |
| **近期** | 2021-2026 | 带年份前缀 `2022-Q1.md` | ADOC | 三项元信息齐全；引言无 `## 引言` 标题 |

## 二、文件命名规范

### 2.1 命名原则

- 文件名不得包含空格、中文字符或英文冒号 `:`
- 必须兼容 Windows 文件系统

### 2.2 命名模式

| 年代 | 命名模式 | 示例 |
| ---- | -------- | ---- |
| 2001-2009 | `M-M.md` 或 `M.md`（月份范围） | `1-3.md`、`6.md` |
| 2010-2016 | `QN.md`（季度） | `Q1.md` |
| 2017 | `M-M.md`（恢复月份范围） | `1-3.md` |
| 2018 | `YYYY-MM-YYYY-MM.md`（跨年特殊命名） | `2017-10-2018-9.md` |
| 2019-2021 | `QN.md`（季度） | `Q1.md` |
| 2022 至今 | `YYYY-QN.md`（带年份前缀） | `2022-Q1.md` |

### 2.3 已知异常（待修正）

- `2024/2024-q1.md` 至 `2024/2024-q4.md`：小写 `q`，应改为大写 `Q`，统一为 `2024-Q1.md` 至 `2024-Q4.md`
  - **注意**：改名需同步更新 `SUMMARY.md`、`mu-lu.md` 中的引用，并执行 `git mv` 保留历史

## 三、元信息头部规范

### 3.1 标准格式（三项齐全）

```markdown
# FreeBSD YYYY 年第N季度状态报告

- 原文：[English Title](https://www.freebsd.org/status/report-YYYY-MM-YYYY-MM/)
- 版本：YYYY 年 M 月 D 日
- 最后作者：Author Name
```

### 3.2 字段规则

| 字段 | 是否必填 | 说明 |
| ---- | -------- | ---- |
| `原文` | 必填 | 统一用"原文"二字，**不**用"原文链接" |
| `版本` | 2021 起必填；2020 及之前若上游无版本信息则可省略 | 日期格式为 `YYYY 年 M 月 D 日`（中文点分隔） |
| `最后作者` | 必填 | 以英文原版页面底部署名为准；若署名为"FreeBSD 项目"则保留 |

### 3.3 URL 后缀规则

- 2021 及以后：URL 以 `/` 结尾（无 `.html`）
- 2020 及之前：URL 保留 `.html` 后缀（历史链接，不强制改）
- **例外**：若 `.html` 链接已失效而 `/` 链接有效，则改为 `/` 后缀（需联网复查三次确认）

### 3.4 早期报告版本字段补全流程

1. 访问对应英文原版页面（URL 见"原文"项）
2. 查找页面底部署名及发布日期
3. 若有明确日期，补全为 `- 版本：YYYY 年 M 月 D 日`
4. 若页面无日期信息，则省略此行（不写"版本：未知"）
5. 联网复查三次后确认

## 四、引言规范

### 4.1 标题

**所有报告统一保留 `## 引言` 标题**，包括 2021 年起取消该标题的近期报告。

### 4.2 引言结构

```markdown
## 引言

（正文段落，每段单独一行，段落间用空行分隔）

Author Name

（职位/团队，若有）

（客套语，若有）
```

### 4.3 署名格式

- **不使用破折号** `——` 引出署名
- 人名、职位/团队、客套语均独立隔行（每项之间用空行分隔）
- 若原文有多行署名（如"Lorenzo Salvadore\n\n代表状态团队敬上。"），保留多行结构，不合并为一行
- **客套语规则**：若原文有客套语（如"敬上"），保留原文客套语；若原文无客套语，则补"此致"
- 若原文无署名，则不添加

**示例**：

```markdown
Lorenzo Salvadore

代表状态团队

敬上
```

**无客套语时的示例**：

```markdown
Lorenzo Salvadore

代表状态团队

此致
```

## 五、章节层级规范

### 5.1 标准层级

```markdown
## 引言
## FreeBSD 团队报告
### FreeBSD 核心团队
### FreeBSD 基金会
#### 子小节（如"筹款工作"、"操作系统改进"）
## 项目
### 子项目A
### 子项目B
## 杂项                ← 仅 2017-10-12 草稿等特殊报告使用
```

### 5.2 章节分组规则

- **所有报告**统一使用 `## FreeBSD 团队报告` 作为团队类条目的分组标题
- 团队条目（核心团队、基金会、集群管理、Ports 管理、文档工程、发布工程、CI 等）归入 `## FreeBSD 团队报告`
- 非团队条目归入 `## 项目` 或 `## 杂项`
- 早期报告（2001-2009）若原文无明确分组，**不强行添加** `## FreeBSD 团队报告`，保持扁平 `## 项目` 结构

### 5.3 团队报告下的引导句

`## FreeBSD 团队报告` 标题下统一使用一句引导语：

```markdown
## FreeBSD 团队报告

来自各官方及半官方团队的报告，见 [管理页面](https://www.freebsd.org/administration/)。
```

**注意**：

- 2020 及之前使用 `https://www.freebsd.org/administration.html`（保留 `.html`）
- 2021 及之后使用 `https://www.freebsd.org/administration/`（无 `.html`）
- 引导句表述统一为“来自各官方及半官方团队的报告，见 [管理页面](URL)。”

## 六、链接区块规范

### 6.1 标准格式（多条链接）

```markdown
### 子项目标题

**链接**：

[文本](URL)

[文本](URL)

**联系方式**：姓名 <[邮箱](mailto:邮箱)>
```

### 6.2 单条链接格式

当区块内只有一条链接时，"**链接**："与 URL 同行，不空行：

```markdown
### 子项目标题

**链接**：[文本](URL)

**联系方式**：姓名 <[邮箱](mailto:邮箱)>
```

### 6.3 规则

- 标题下方的"链接"二字**加粗**，后接中文全角冒号 `：`（冒号在加粗标记**外面**，冒号后不空格，与 `**联系方式**：` 格式一致）
- **单条链接**：`**链接**：[文本](URL)` 同行，不空行
- **多条链接**：`**链接**：` 单独成行，下方空一行后列出各 URL，每个 URL 独占一行，URL 之间用空行分隔
- URL 统一使用 markdown 链接格式 `[文本](URL)`
- **早期报告（2001-2009）的裸 URL（如 `<http://...>`）保留原样**，不强制改为 markdown 链接；单条裸 URL 时为 `**链接**：<http://...>`
- 若区块内无链接，则不写"链接"行

### 6.4 异常修正

- `链接`（未加粗）→ `**链接**：`
- `链接：`（未加粗带冒号）→ `**链接**：`
- `**链接**`（无冒号）→ `**链接**：`

## 七、联系方式规范

### 7.1 标准格式

```markdown
**联系方式**：姓名 <[邮箱](mailto:邮箱)>
```

### 7.2 规则

- "联系方式"四字**加粗**
- 冒号使用中文全角冒号 `：`
- 邮箱用 markdown 链接包裹，外层用尖括号 `< >`
- 多个联系人时，每个联系人单独一行
- 若联系人为团队（如"FreeBSD 核心团队"），团队名后接 `<邮箱链接>`

### 7.3 异常修正

- `联系方式：`（未加粗）→ `**联系方式**：`（加粗）
- `**联系方式：**`（冒号在加粗内）→ `**联系方式**：`（冒号移出加粗）
- `联系方式：姓名 [邮箱](mailto:邮箱)`（无尖括号）→ `**联系方式**：姓名 <[邮箱](mailto:邮箱)>`
- `联系方式：姓名（[邮箱](mailto:邮箱)）`（圆括号）→ `**联系方式**：姓名 <[邮箱](mailto:邮箱)>`

## 八、待办事项规范

### 8.1 标准格式

```markdown
#### 待办事项

1. 第一项
2. 第二项
3. 第三项
```

### 8.2 规则

- 标题统一为 `#### 待办事项`（四级标题）
- 使用有序列表，编号连续
- 若原文无待办事项，则不添加

## 九、提交权限小节规范

### 9.1 标准格式

```markdown
#### 提交权限

- 核心团队批准了 XX (xx@) 的 src 提交权限
- 核心团队批准恢复 YY (yy@) 的 src 提交权限
```

### 9.2 规则

- 标题统一为 `#### 提交权限`（中文"提交"，**不**用英文"Commit"）
- 使用无序列表
- 用户名后的提交权限标识（如 `src`、`ports`、`doc`）保持小写

## 十一、引言署名与署名行规范

### 11.1 引言末尾署名

引言正文最后若有人署名，统一格式（不使用破折号，各项独立隔行）：

```markdown
Author Name
```

- **不使用破折号** `——`
- 人名、职位/团队、客套语均独立隔行（每项之间用空行分隔）
- 署名区块作为独立段落，前后用空行分隔

### 11.2 多人署名

若原文为多人署名，用顿号分隔：

```markdown
Author A、Author B
```

### 11.3 代表团队

若原文含"代表状态团队"等表述，保留多行结构：

```markdown
Author Name

代表状态团队

敬上
```

**客套语规则**：若原文有客套语（如"敬上"），保留原文客套语；若原文无客套语，则补"此致"。客套语保留在独立行，不省略。

## 十二、URL 与链接规范

### 12.1 正文链接

- 正文中的 URL 统一使用 markdown 链接 `[文本](URL)`
- **早期报告（2001-2009）的裸 URL 保留原样**，不强制改造
- 中期及以后报告中的裸 URL 改为 markdown 链接

### 12.2 邮箱链接

- 邮箱统一使用 `[邮箱](mailto:邮箱)` 格式
- 在"联系方式"行中，邮箱外层用尖括号 `< >` 包裹

### 12.3 链接稳定性

- 修改 URL 前需联网复查三次确认链接有效
- 若原 URL 失效，改为有效的等价 URL（优先 FreeBSD 官方域名）
- 不破坏带锚点的链接（如 `#ETHERNET`）

## 十七、格式规范

本节汇总全书正文与代码格式规则，供校对时逐项核对。

### 17.1 man 页引用

- 带括号数字的 man 页引用（如 `vt(4)`、`netmap(4)`、`xz(1)`、`jail(8)`、`make(1)`、`acpiconf(8)`、`makefs(8)`、`fortune(6)` 等）一律使用 **加粗**，**不**使用反引号
- **禁止**将 `vt(4)` 改为 `vt.4` 等点号格式，必须保持 `命令(N)` 原始格式
- 带链接的 man 页引用（如 `[xxx(n)](url)` 格式）**维持原样**，不加粗、不加反引号、不破坏链接

**正确示例**：

```markdown
**vt(4)** 驱动、**make(1)** 命令、参考 **jail(8)**
```

**错误示例**：

```markdown
`vt(4)` 驱动、vt.4 驱动、`make(1)` 命令
```

### 17.2 命令、选项、参数

- 命令、选项、可调选项、可调参数等格式使用 `行间代码`（反引号）包裹
- **不带选项或参数的裸命令**（如 ls、cd、mv、cp、mkdir、rm、ps、cat、grep、sed、awk 等）**不加反引号、不进行任何包裹**，保持原样
- 带选项或参数的命令（如 `ls -l`、`mv file1 file2`、`pkg install`）仍使用反引号包裹

**正确示例**：

```markdown
使用 `pkg install net/ceph-devel` 安装。
运行 `ls -l` 查看；用 ls 列出文件。
```

### 17.3 路径与 IP 地址

- 正文中的路径（带 `/` 或 `\` 的，如 `/etc/rc.conf`、`/usr/local/etc/`）和 IP 地址一律使用 **加粗**
- **不**使用反引号 `` ` `` 包裹路径与 IP 地址
- **不**混合使用 `*` 和 `` ` ``（如 `` *`path`* `` 是错误的）

**正确示例**：

```markdown
编辑 **/etc/rc.conf** 文件。
访问 **192.168.1.1**。
```

### 17.4 转义字符

- 含转义字符 `\` 的元素（非命令、选项、参数）一律使用 **加粗** 包裹整个元素
- **不**在正文中直接使用转义字符（如写 **PROTO_TYPE** 而非 `PROTO\_TYPE`）
- 逐个手动修改，**禁止批量替换**

### 17.5 引号规范

- **正文**：禁止使用日式引号「」『』，一律使用中文双引号“”或单引号‘’
- **正文**：双引号使用“”，不使用半角 "" 或 ''
- **正文**：标点符号统一使用全角，包括“”、括号、引号等
- **代码块非注释部分**：所有引号必须使用半角 "" 或 ''（代码语法需要）
- **代码块注释部分**：可使用“”或‘’
- 正文中引用系统错误消息、命令输出等代码内容时，应使用反引号包裹，使内部半角引号成为代码内容
- 逐个手动修改，**禁止批量替换**

### 17.6 加粗与空格

- `汉字 **纯粹中文文字加粗内容** 汉字` 中间的加粗内容前后必须有一个空格（前后有标点则不计入此条）
- 避免“xx yy”类倒置结构，如“xx 可调参数”应改为“可调参数 xx”

### 17.7 代码块

- 无法判断语言标记时，使用 ` ```sh ` 兜底
- **禁止**使用 ` ```text ` 作为代码块标记
- **不窜改**既有的 ` ``` ` 或 ` ```ini ` 标记
- 代码块内的英文注释**必须翻译为中文**（如 `# This is a comment` → `# 这是一个注释`）
- 只翻译注释部分，**不修改**实际命令或代码
- 保持代码结构和格式不变
- **fstab 文件表头**（如 `# Device Mountpoint FStype Options Dump Pass#`）及相关内容保持英文原样，**不翻译**

### 17.8 提示块

- tip / important / note / warning / caution 使用 `>` 缩进引用
- 关键词 **加粗**（如 **提示**、**重要**、**注意**、**警告**、**小心**）

### 17.9 表格

- 表格一律**居中**
- 表头声明几列，所有数据行都必须有几列
- 末尾缺空单元格时需补 `| |`，**不**省略管道符
- compact 风格：管道符两侧各保留**单个空格**，空单元格写作 `| |`

### 17.10 HTML 与 AsciiDoc 残留

- 本项目**不支持任何 HTML 语法**
- **禁止**在 Markdown 中保留 AsciiDoc 的 `::` 定义列表语法
- **禁止**保留 `+` 续行标记
- AsciiDoc 定义列表术语（如 `dvd1::`、`disc1::`、`bootonly::`、`memstick::`、`mini-memstick::`、`FreeBSD/arm SD 卡镜像::`）必须转换为 Markdown 加粗标签格式：`**术语**` + 空行 + 描述文本
- `+` 续行标记必须移除，段落间用空行分隔

### 17.11 段落换行

- 一段内容必须**在同一行**，不同段必须换行
- 段落之间用**空行**分隔
- **禁止**在同一段落内插入软换行（两行普通文本之间无空行分隔却属于同一段落的情况）

### 17.12 CJK 空格

- 中英文、中文与数字之间**必须加半角空格**
- 文件和命令名用反引号括起来

### 17.13 禁止事项

- **禁止篡改**软件版本号、用户名、带圈数字（如 ①②③），确保其位置、数量与英文原文一致
- **禁止**章节交叉引用（如"参见第 5 章"），改用语义化链接
- 英文原文中的 HTML 锚点（如 `#ETHERNET`、`#BLUETOOTH`、`#INTRO`）必须改为指向 FreeBSD 官方网站对应页面的外链
- **避免滥用“已”字**：如"XX 已新增"应改为"新增 XX"
- **避免滥用“一个 xx”**：如"一个驱动"应改为"某驱动"或具体名称
- **避免欧化汉语**：倒装句、后置句、偷换主语、不必要的被动句应改为地道汉语表述
